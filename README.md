# BVM Manual Inspection Station

The BVM Manual Inspection Station is an industrial application designed to guide operators through manual measurement tasks using real-time instructional videos and structured data capture.

The system assists operators during inspection workflows by combining step-by-step video guidance with measurement logging and historical tracking.

## System Architecture

The application follows a cloud-deployed split architecture consisting of three main components:

### Frontend
A Flutter Web application hosted as a static site on Amazon S3. The frontend provides the operator interface and communicates with the backend API.

### Backend
A FastAPI application containerized using Docker and deployed on Amazon ECS. The backend handles:
- Measurement data submission
- Operator session management
- Instructional video retrieval
- Export generation

### Storage
Two Amazon S3 buckets are used:
- **pcbias-frontend**: Hosts the compiled Flutter web application.
- **pcbias-videos**: Stores instructional videos and exported measurement files.

The videos bucket is private, and video access is provided through temporary presigned URLs generated by the backend.

## Cloud Deployment Overview

The system is deployed using a cloud-native architecture built on AWS services.

### Frontend Hosting
The Flutter application is compiled into static assets and hosted on Amazon S3 static website hosting.

### Backend Deployment
The FastAPI backend is:
- Containerized using Docker
- Stored in Amazon Elastic Container Registry (ECR)
- Deployed as a service using Amazon ECS

### Secure Media Delivery
Instructional videos are stored in a private S3 bucket. The backend generates presigned URLs allowing the frontend to temporarily access video files without exposing the bucket publicly.

### CI/CD Automation
Deployment pipelines are implemented using GitHub Actions:
- **Frontend pipeline**: Builds Flutter Web and syncs the build output to the S3 frontend bucket.
- **Backend pipeline**: Builds the Docker image, pushes it to ECR, and triggers ECS service updates.

## Workflow

The application follows a structured inspection workflow to maintain data integrity.

### 1. Operator Sign-in
Users enter their name and roll number to initialize a session.

### 2. Hardware Setup and Calibration
The system checks connectivity with measurement devices such as digital calipers and performs calibration if required.

### 3. Category Selection
Operators choose between measurement categories: Housing or Shaft.

### 4. Inspection Process
Operators perform the inspection in guided steps:
- Enter the Product ID
- Watch instructional video guidance
- Capture the required measurements
- Input measurement values into the system

Each step ensures that the correct measurement procedure is followed.

### 5. Data Submission
Once all measurements are completed, the operator reviews the values and submits the inspection data.

### 6. Measurement History
Operators can view historical measurements and export them when needed. Exports are generated dynamically and stored temporarily in S3.

## Static Web Hosting

The frontend is optimized for static deployment.

### Build Process
To generate the web distribution files:
```bash
cd frontend
flutter build web --release
```
The compiled files are generated in `frontend/build/web/`.

### Deployment
The compiled web assets are deployed to an Amazon S3 bucket configured for static website hosting. Deployment is automated through GitHub Actions.

## Backend Deployment

The backend service runs as a containerized FastAPI application.

### Container Build
The backend is packaged into a Docker image. Example:
```bash
docker build -t inspection-api .
```

### Image Registry
Container images are stored in Amazon Elastic Registry (ECR).

### Runtime Environment
The container is deployed through Amazon ECS, which manages:
- Container orchestration
- Service scaling
- Task lifecycle
- Logging through CloudWatch

### Secure Media Access
Instructional videos are stored in a private S3 bucket. The backend generates presigned URLs to allow temporary access for the frontend. This ensures:
- The bucket remains private
- Video access is time-limited
- No public exposure of media assets

### Exported Measurement Files
Measurement exports are generated on demand. These files:
- Are temporarily stored in S3
- Are accessible via presigned URLs
- Are automatically deleted using S3 lifecycle policies

## Development Setup

### Backend
Install dependencies:
```bash
pip install fastapi uvicorn pandas boto3
```
Run the development server:
```bash
uvicorn app.main:app --reload
```

### Frontend
Install dependencies:
```bash
flutter pub get
```
Run the development server:
```bash
flutter run -d chrome
```

## Environment Configuration
The frontend backend endpoint is injected during build using:
```bash
--dart-define=BACKEND_URL=<backend_url>
```
Example:
```bash
flutter build web --dart-define=BACKEND_URL=https://api.example.com
```
